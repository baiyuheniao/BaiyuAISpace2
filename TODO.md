## 插件系统

现有的功能扩展点（MCP、Skill）都是应用内置的固定形态，第三方开发者没有更底层的接入方式。
考虑做一套插件系统，让开发者能给应用扩展新能力（自定义工具、自定义 UI 面板等），而不只是
通过 MCP 协议接入外部工具。

需要想清楚的点：
- 插件加载机制（本地目录扫描 / 手动导入包），要不要签名校验
- 权限沙箱：插件可能需要访问文件系统、网络，怎么限制范围、怎么向用户展示"这个插件能做什么"
- 和现有 MCP/Skill 体系的边界——插件是否只是"给开发者的更底层扩展点"，普通用户仍然只接触
  MCP 市场/Skill 这类高层封装

## 新手引导

目前首次打开应用没有任何引导流程，新用户直接看到空的对话页（`ChatView.vue` 提示"请先前往
设置创建 API 配置"），需要自己摸索"先去设置配 API"这类前置知识，也不知道 MCP/Skill/Agent
Team 这些概念是什么、什么时候该用。

考虑做一个首次启动的分步引导，覆盖：配置 API → 简单介绍 MCP/Skill/Agent Team 各自是什么、
分别解决什么问题 → 引导发出第一条消息。引导本身的视觉风格要遵循黑白编辑设计系统（见
`design-system` skill），不能用高亮色块这类和整体调性冲突的常见引导库样式。
