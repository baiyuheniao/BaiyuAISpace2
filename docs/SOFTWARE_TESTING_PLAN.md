## 总体测试策略

### 1.1 测试目标

- **功能完整性**：验证所有声明的功能是否按需求工作
- **用户体验**：评估界面友好性、响应速度和易用性
- **数据安全性**：验证 API 密钥加密存储、隐私保护机制
- **跨平台兼容性**：在 Windows、macOS、Linux 上运行无误
- **边界情况处理**：测试异常输入、网络异常、资源限制场景
- **性能指标**：确保启动速度、消息响应、数据库查询在可接受范围内

### 1.2 测试范围

| 模块 | 范围 | 优先级 |
|------|------|--------|
| **对话 & LLM** | 多提供商接入、流式输出、中断、会话管理 | 🔴 P0 |
| **知识库 (RAG)** | 文档导入、向量化、检索、删除操作 | 🔴 P0 |
| **MCP 工具** | 服务器添加、工具发现、函数调用、预设安装 | 🟠 P1 |
| **Skill 管理** | 创建、编辑、删除、预设导入 | 🟠 P1 |
| **Agent Team** | Agent 生命周期、消息路由、圆桌会议 | 🟠 P1 |
| **定时任务** | 任务创建、调度类型、触发机制、状态管理 | 🟡 P2 |
| **本地部署** | Ollama/LM Studio 集成、模型管理 | 🟡 P2 |
| **设置 & 历史** | 配置保存、偏好设置、历史查询 | 🟡 P2 |

### 1.3 测试方法

- **功能测试（Function Test）**：黑盒测试，验证功能符合需求
- **界面测试（UI Test）**：检查外观、布局、文本、按钮响应
- **交互测试（Interaction Test）**：验证点击、输入、滚动、快捷键等
- **负面测试（Negative Test）**：异常输入、边界值、错误条件
- **压力测试（Stress Test）**：大数据量、长时间运行、并发操作
- **兼容性测试（Compatibility Test）**：跨操作系统、浏览器版本（如适用）
- **安全测试（Security Test）**：输入验证、认证、数据加密、敏感信息处理

## 测试环境准备

### 2.1 硬件环境

| 项目 | 最低配置 | 推荐配置 |
|------|---------|---------|
| **CPU** | 双核 2.0 GHz | 四核 2.5 GHz+ |
| **内存** | 8 GB | 16 GB+ |
| **存储** | 2 GB 可用空间 | 10 GB+ |
| **网络** | 10 Mbps | 100 Mbps+ |

#### Windows 环境
- **OS**：Windows 10 (Build 19041+) 或 Windows 11
- **Node.js**：18.16.0 或更高版本
- **pnpm**：8.0 或更高版本
- **Rust**：1.75 或更高版本
- **Visual C++ 生成工具**：2019 或更新版本
- **Git**：2.40 或更新版本

#### macOS 环境
- **OS**：macOS 10.15 (Catalina) 或更高版本
- **Xcode Command Line Tools**：最新版本
- **Node.js**：18.16.0 或更高版本
- **pnpm**：8.0 或更高版本
- **Rust**：1.75 或更高版本

#### Linux 环境
- **OS**：Ubuntu 20.04 LTS 或兼容发行版
- **build-essential**：最新版本
- **Node.js**：18.16.0 或更高版本
- **pnpm**：8.0 或更高版本
- **Rust**：1.75 或更高版本

### 2.3 环境搭建步骤

**第一步**：克隆仓库
```bash
git clone https://github.com/baiyuheniao/BaiyuAISpace.git
cd BaiyuAISpace2
```

**第二步**：安装依赖
```bash
pnpm install
```

**第三步**：获取测试 API 密钥
- OpenAI API Key：https://platform.openai.com/account/api-keys
- Anthropic API Key：https://console.anthropic.com/
- 其他提供商：参考 docs/api-manuals/ 目录

**第四步**：启动开发环境
```bash
pnpm tauri dev
```

或直接运行已构建的版本（如有）。

- 目标：验证应用基本可用性

- 用例：TC-LLM-001, TC-LLM-002, TC-KB-001, TC-UI-001
- 结果：绿灯或停止进一步测试

**第 2 天 - P0 功能测试**
- 目标：详细测试所有 P0 优先级用例
- 时间：4-6 小时
- 用例：所有 P0 标记的 TC
- 结果：绿灯或列出关键缺陷

**第 3 天 - P1 功能测试**
- 目标：测试 P1 功能
- 时间：4-6 小时
- 用例：所有 P1 标记的 TC
- 结果：记录所有发现的缺陷

**第 4 天 - 界面与交互测试**
- 目标：详细测试 UI/UX
- 时间：3-4 小时
- 用例：TC-UI-* 系列
- 结果：列出设计问题和布局缺陷

**第 5 天 - 性能与兼容性测试**
- 目标：验证性能指标和跨平台兼容性
- 时间：4-6 小时
- 用例：TC-PERF-*、TC-COMPAT-* 系列
- 结果：性能报告、兼容性矩阵

**第 6 天 - 安全性与稳定性测试**
- 目标：深入测试安全性、长时间稳定性
- 时间：4-6 小时
- 用例：TC-SEC-*、TC-PERF-013 到 TC-PERF-018
- 结果：安全评估报告

**第 7 天 - 缺陷验证与回归测试**
- 目标：验证已修复的缺陷，确保无新缺陷
- 时间：2-3 小时
- 用例：涉及修复的 TC，以及相邻功能
- 结果：最终缺陷列表、版本发布建议

### 9.3 缺陷修复与重测流程

```
发现缺陷
    ↓
记录缺陷报告（优先级、描述、重现步骤）
    ↓
分配给开发人员
    ↓
开发人员修复
    ↓
测试人员重新测试（原缺陷 + 相邻功能）
    ↓
验证通过 → 标记已修复
│
└→ 验证失败 → 提出问题、重新分配
```

### 9.4 测试报告与发布决策

**测试报告包含**：
- 执行摘要（测试覆盖率、缺陷统计）
- 缺陷列表（按优先级分类）
- 性能指标汇总
- 兼容性矩阵
- 安全评估结论
- 发布建议

**发布决策标准**：
- ✅ **可发布**：无 P0 缺陷，P1 缺陷 < 5 个，已知问题记入发行说明
- ⚠️  **有条件发布**：P0 缺陷已计划修复，P1 缺陷不超过 10 个
- ❌ **延迟发布**：P0 缺陷 > 1 或基本功能不可用

### B. 常见问题排查表

| 问题 | 排查步骤 | 解决方案 |
|------|---------|---------|
| 应用启动慢 | 1. 检查启动时间 2. 查看 CPU/内存使用 3. 检查网络连接 | 清除缓存、关闭其他进程、升级硬件 |
| API 连接失败 | 1. 验证 API Key 正确性 2. 检查网络连接 3. 验证端点地址 | 重新输入 Key、检查防火墙、验证端点 |
| 知识库检索无结果 | 1. 确认文档已导入 2. 检查索引进度 3. 验证查询词 | 重新导入文档、手动重新索引、调整查询 |
| 应用崩溃 | 1. 查看日志文件 2. 记录重现步骤 3. 收集系统信息 | 升级应用、清除数据、提交缺陷报告 |
